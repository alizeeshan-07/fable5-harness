# Fable Loop (Adaptive) Skill

Use this skill when the user invokes `/fable-loop-adaptive`. It converts one request into a **self-scoring convergence loop** in which the evals are *generated by the model for the specific artifact*, rather than picked from a fixed checklist.

Core principle: the prompt is only the map; the real work is the territory — and you don't get to declare the territory conquered on your own say-so. So the skill (1) works out what "good" means *for this exact artifact*, (2) turns that into a fixed, scored eval set, and (3) iterates against it autonomously, showing its scores each pass, until the score stops climbing.

**What makes this variant different:** it does not hardcode which evals to use. It hardcodes *how to generate good ones* (a method) and *what good looks like* (a small set of calibration exemplars). This keeps it general across any domain — listed or not — while the method and the [V]/[S] discipline hold the quality floor. Full guidance is in `reference/eval-generation.md`.

## The one-shot promise

After a single setup exchange, the skill runs the whole loop itself:

```
sense domain -> GENERATE top ~10 evals for this artifact -> user sets iterations + target
             -> [ build -> score -> diagnose -> refine ] x N   (autonomous)
             -> final scorecard
```

The only mid-flight interruptions are decisions the skill may not make alone: a high-risk irreversible choice, or an eval it genuinely cannot score without the user.

## Invocation behavior

- `/fable-loop-adaptive <goal>` — run the full autonomous flow.
- `/fable-loop-adaptive` — ask once for the goal, then run.
- `/fable-loop-adaptive domain-scan` — sense the domain only.
- `/fable-loop-adaptive evals` — generate (or regenerate) the top-10 eval set and stop.
- `/fable-loop-adaptive run` — start/continue the scored loop.
- `/fable-loop-adaptive rescore` — re-score the current artifact against the frozen set.
- `/fable-loop-adaptive freeze` — save the current eval set for reproducible re-use (see Reproducibility).
- `/fable-loop-adaptive reuse` — load a previously frozen eval set instead of generating a new one.
- `/fable-loop-adaptive report` — emit the final scorecard.

If `$ARGUMENTS` is empty or vague, ask one question to get the goal, then proceed.

## Non-negotiable behavior — loop integrity

A self-scoring loop is worthless if the scorer inflates its own grades, and a self-*generated* eval set is worthless if it is vague and self-flattering. These rules are what make both the evals and the score mean something. Reasoning is in `reference/eval-and-loop.md`.

1. **Generate discriminating, verifiable-biased evals.** When you create the eval set, favor checks that (a) separate good work from bad and (b) can be verified by running something. A set full of soft "is it good? 5/5" evals is a failed set — regenerate it. This is the main risk of letting the model author its own evals; the method in `reference/eval-generation.md` exists to counter it.
2. **Freeze the rubric before the loop.** Once approved, the eval set, weights, and scales are fixed. Never edit an eval to make failing work pass. A genuine rubric bug is fixed in the open with the user and never counts as a score gain.
3. **Prefer verifiable scorers; label the rest.** Run real checks (tests, build, lint, metric, script) and use the result. Tag every eval **[V] verifiable** or **[S] self-judged**; the final scorecard reports the point split so the user knows how much to trust the number.
4. **Score to lose points.** Grade adversarially; every score carries one-line evidence; a total rises only when a concrete change justifies it.
5. **Never fake monotonic improvement.** A plateau below target is an honest, reportable result. Do not nudge scores to manufacture progress.
6. **Show every pass** — per-eval scores, weighted total, delta.
7. **Diagnose before refining** — one-line root-cause hypothesis for the lowest evals.
8. **Respect the budget and stop rules** — stop at target, cap, 2-iteration plateau, or high-risk unknown.
9. **Conservative on unknowns** — reversible/low-risk: log and continue; irreversible/high-risk: stop and ask.

## Phase 1 — Sense the domain

From the goal plus any files/links/context, infer where the user is working and what they're producing. The domain is *orientation for generation*, not a category to pull a fixed list from. Output:

```md
## Domain scan
Goal (restated): ...
Domain: [best-fit label, or "novel — deriving from first principles"]
Artifact type: [what will actually be produced]
What "good" means here: [2–4 axes that matter most for this specific artifact]
Assumptions I'm making: [so the user can correct them]
```

If the domain/artifact is ambiguous in a way that changes the evals, ask exactly one question. Otherwise proceed — the eval-approval gate is where the user corrects you.

## Phase 2 — Generate the evals (from the method, for this artifact)

Derive the ~10 highest-signal evals for *this specific artifact* using the generation method in `reference/eval-generation.md` (the lenses: correctness, robustness, constraints, fit, craft, safety, cost). Do not select from a menu — the exemplars in that file are calibration for the *shape and quality bar* only; always tailor to the actual work. Rank by discriminating power and verifiability, weight to sum 100, score each on 0–5, and tag [V]/[S]. Always name the domain's **verifiability ceiling** — the real outcome your evals cannot directly measure (e.g. deliverability, ranking, reply rate) — so the score is not mistaken for the outcome.

Present the generated set as an editable table:

```md
## Generated evals (frozen once you approve)
| # | Eval | What it measures | Scorer | V/S | Weight | Scale |
|---|------|------------------|--------|-----|--------|-------|
| ... (up to ~10, weights sum to 100) |
Verifiable coverage: X of 10 are [V], carrying Y of 100 weight.
Verifiability ceiling: [the real outcome this cannot measure].
```

## Phase 3 — Setup gate (the only required input)

Ask the user, in one message, to confirm:

1. **Evals** — keep / drop / edit / reweight any of the generated 10.
2. **Max iterations** — hard cap (default 3–5).
3. **Target score** — stop-early threshold on 0–100, or "converge" (default).

Optionally offer: *freeze this set for reuse?* (see Reproducibility). Once they answer, restate the frozen set + budget and start the loop. This is the last routine prompt needed.

## Phase 4 — The scored convergence loop (autonomous)

Track with `templates/scorecard.md`. Announce the budget, then loop. Each iteration: build/refine the smallest change targeting the lowest evals; score against the frozen set (run [V] evals for real, apply rubrics for [S] adversarially, one-line evidence each); compute the weighted total (0–100, formula in `reference/eval-and-loop.md`); emit the iteration scorecard; decide via stop rules (target / cap / plateau / risk / verified-max). Never rewrite the rubric to move the number.

```md
### Iteration i/N — total: 78/100 (Δ +12) | verified 55/70 | self-judged 23/30
| # | Eval | Score | V/S | Evidence |
Lowest: #4 (2/5), #7 (2/5). Hypothesis: ...
Decision: continue / stop-target / stop-plateau / stop-cap / stop-risk / stop-verified-max
```

## Phase 5 — Final scorecard and report

Deliver the artifact plus a report from `templates/scorecard.md` and `templates/explainer-and-quiz.md`: best iteration and total, iterations run, stop reason, trajectory, final per-eval table, a **trust note** stating verified vs self-judged points, residual weaknesses, and what would raise the score further. Close with a 3–5 question quiz (including which eval would catch a regression). Offer the outer loop: fold new findings into the frozen set (every bug becomes a permanent regression eval) and run another cycle.

## Reproducibility — freeze and reuse

Generation is dynamic by default, which means two runs on the same task can produce different eval sets, so scores are not comparable across runs. When the user wants comparable, repeatable scoring:

- **Freeze** (`freeze`): persist the approved eval set — weights, scales, scorer definitions, and [S] rubrics — to `templates/eval-spec.md` (or a project file the user names). This is the canonical, frozen rubric.
- **Reuse** (`reuse`): load that frozen set and score against it verbatim instead of generating a new one. Same rubric across runs = comparable scores and honest trend tracking.
- **Regenerate** only when the artifact or goal changes enough that the frozen set no longer fits — and say so explicitly, because it breaks comparability with prior runs.

Use dynamic generation for exploration; freeze once the eval set is trusted and you want to measure improvement over time.

## Style

Keep each pass legible and terse: iteration, change, score, delta, decision. Lead the setup gate with the eval table. Be honest about the score's limits — a defensible 82 with evidence beats a fabricated 100.
